# wasm-vm: The Road to the Browser Singularity

**A Kardashev scale for in-browser Linux.**

This document is the guiding trajectory for `wasm-vm`: a from-scratch, Rust-implemented,
WASM-compiled virtual machine that boots real, unmodified Linux in a browser tab — the
end state being full parity with (and then transcendence of) https://webvm.io/alpine.html:
a persistent, networked, GUI-capable Linux system that lives at a URL.

Like Kurzweil's law of accelerating returns, each level here is not just "more features" —
each level is a *phase change* that makes the next level cheaper to reach. The interpreter
makes the CPU debuggable; the compliant CPU makes Linux bootable; booted Linux makes the
system self-testing (the guest becomes our test harness); the JIT makes compilers usable
inside the guest; and at the top (cancelled 2026-07-06), the system was to run *stock
Chromium* and make it
time-travelable. That closed loop is the singularity condition.

---

## Prior art & where this project sits

- **webvm.io (CheerpX)** — x86 binary translation to WASM, closed-source engine. The target
  experience to clone.
- **JSLinux / TinyEMU (Bellard)** — proof that RISC-V system emulation boots Linux in JS/WASM.
  The architectural blueprint.
- **v86** — x86 interpreter+JIT in Rust/WASM. Proof the JIT-in-browser approach works.
- **container2wasm** — Bochs/TinyEMU-based container boot in WASM.
- **box64 (ptitSeb)** — an x86_64→native userspace translator. Our path to running x86_64
  Linux binaries *inside* the RISC-V machine (Layer F / Epic 7) — CheerpX-via-emulation.
- **`~/Dev/almostnode`** — our own prior work: Node.js API shims, virtual FS, mock OS UI
  (`apps/almost-os`), web IDEs. That is the *emulation-of-APIs* approach. `wasm-vm` is the
  *emulation-of-hardware* approach: no shims, no mocks — a machine. The almostnode apps
  become candidate front-ends/shells for this VM at Level 5+.

## The three irreversible architectural bets

1. **Guest ISA: RISC-V (RV64GC), not x86.** CheerpX spent years on x86 translation. RV64GC
   is a clean, fully-open, formally specified ISA with an official compliance suite
   (riscv-tests, RISCOF), and Alpine Linux ships riscv64 as a release architecture. Every
   binary webvm runs has an Alpine riscv64 equivalent. This single decision cuts the CPU
   effort by ~10x. (x86_64 support arrives *inside* the machine at Layer F via box64 — a
   guest-side userspace translator, not a second CPU core.)
2. **Language/runtime: Rust → `wasm32`, no_std-friendly core.** The emulator core is a pure
   Rust crate with zero web dependencies (testable natively at native speed), wrapped by a
   thin `wasm-bindgen` boundary. Everything browser-specific (IndexedDB, WebSocket, canvas,
   WebAudio, workers) lives behind Rust traits so the core never knows it's in a browser.
3. **Device model: virtio everywhere.** Block, network, GPU, input, sound, filesystem — all
   virtio-mmio. One transport, one ring-buffer implementation, amortized across every device.
   The Linux kernel already has mature drivers for all of them; we never write a guest driver.

## Execution doctrine

- All work is decomposed into tasks under `tasks/`, organized into the Epics below.
- A **priority queue** (`tasks/QUEUE.md`, regenerated by `tools/build_queue.py`) totally
  orders all tasks. Work proceeds one task at a time, top of queue first.
- Every task ends with **adversarial verification**: a verifier whose explicit mission is to
  *refute* the task's completion claim (see `tasks/README.md` for the protocol). A task is
  done when a hostile verifier fails to break it, not when its implementer says it works.
- Each Epic ends in a **capstone task** — a demonstrable, user-visible threshold naming a
  concrete, runnable milestone. You cannot start Epic N+1's capstone-dependent work until
  Epic N's capstone survives adversarial verification.

# The Milestone Ladder

Every epic past the machine ladders toward a **named, runnable target** — a specific piece
of software you can point at and watch run. Each epic also lights up one **capability layer**
(A–G); the layers stack, and each layer is what makes the next target reachable.

```
Level 0  IGNITION        a Rust machine skeleton executing RISC-V in a browser tab
Level 1  THE MACHINE     RV64GC CPU + Sv39 MMU + traps + timers/interrupts        [Layer A]
Level 2  FIRST LIGHT     devices + interrupts + SBI → boot xv6, then Linux to init [Layer B]
Level 3  CIVILIZATION    persistence + net + a real userland → busybox+QuickJS+Node [Layer C]
Level 3.5 OCI WORKLOADS  pull + run real riscv64 containers, cached in the browser   [Layer C+]
Level 4  ACCELERATION    JIT-to-WASM + FENCE.I i-cache coherence → fast Node.js/Bun [Layer D]
Level 5  THE WINDOW      framebuffer/GPU + input + compositor → GUI apps, a surface [Layer E]
Level 6  TRANSCENDENCE   SMP, WebGPU-3D, shareable snapshots, self-hosting        [Layer G*]
Level 7  BABEL           box64 + full network + persistence → x86_64 bins, a desktop [Layer F]
Level 8  CHROME IN CHROME (CANCELLED 2026-07-06) — see the Level 8 section
```

`*` **Layer G (determinism & record/replay) is cross-cutting**, not a level of its own — it
matures from Level 1 onward (see the dedicated section below) and was to reach its capstone at
Level 8 (now cancelled). Level 6 is where its *snapshot/share/time-travel* machinery is
hardened.

## Capability stack — Layers A–G

| Layer | Name | Built in | What it is | What it unlocks |
|-------|------|----------|------------|-----------------|
| **A** | The Machine | E1 | RV64GC CPU, Sv39 MMU, traps, privilege modes, CLINT/PLIC | Execute any bare-metal RISC-V program deterministically |
| **B** | The Platform | E2 | Device tree, SBI firmware, 16550 UART, virtio-blk/net, timers/interrupts | Boot an OS kernel — **xv6**, then a real **Linux** kernel |
| **C** | Userland / ABI | E3 | A real rootfs (musl/glibc), process model, mmap/W^X, threads/futex, package install, networking | Run real userspace: **busybox shell**, **QuickJS**, interpreted **Node.js** |
| **D** | Warp Drive (JIT) | E4 | Dynamic binary translation to WASM, block chaining, FENCE.I i-cache coherence for guest self-modifying code | **Interactive/fast Node.js** (and **Bun**, gated on JSC's weaker RISC-V JIT) |
| **E** | The Surface | E5 | virtio-gpu framebuffer, virtio-input, sound, a compositor (Wayland/X) | **GUI apps** + a **browser rendering surface** |

# The Scale

## Level 0 — IGNITION *(the substrate)*

**Thesis:** Before a machine can think, it must have a body. Level 0 builds the skeleton:
the workspace, the memory bus, a minimal RV64I interpreter, the trace/debug organs, the
native + browser test harnesses, and the adversarial-verification machinery itself.

**What this level gets you:** A Rust core crate that executes bare-metal RISC-V test
binaries identically under `cargo test` (native) and in a browser tab (WASM), with an
instruction-level trace you can diff against a reference emulator (QEMU/Spike). From this
point forward, every bug is *observable* — nothing later is built on an undebuggable base.
This is also where **Layer G is seeded**: the trace hook (E0-T16) and snapshot module
(E0-T17) that record/replay and time-travel are built on already exist here.

**Capstone threshold:** A browser page loads the WASM module, executes a compiled bare-metal
RISC-V program that prints "Hello from RV64" through a stub console, and the instruction
trace matches Spike's, byte for byte.

## Level 1 — THE MACHINE *(a complete CPU — Layer A)*

**Thesis:** Compliance is non-negotiable. Linux does not boot on a CPU that is 98% correct.
Level 1 turns the interpreter into a full RV64GC machine: I/M/A/F/D/C extensions, M/S/U
privilege modes, CSRs, precise exceptions and interrupts, the Sv39 MMU with TLB, CLINT
(timers/IPIs) and PLIC (external interrupts).

**What this level gets you:** A CPU that passes the official riscv-tests and RISCOF
architectural compliance suites. This is the difference between "an emulator that runs my
demos" and "hardware." After Level 1, when Linux misbehaves, the bug is in our devices or
firmware — never silently in the CPU. That certainty is what makes Level 2 a finite task.
**The immediate reward on the other side of this gate is a real OS kernel: xv6-riscv, which
needs only Layer A plus a minimal slice of Layer B** — so xv6 straddles the E1/E2 boundary.

**Capstone threshold:** Green across riscv-tests (rv64ui/um/ua/uf/ud/uc, mi/si) and a RISCOF
run, in both native and WASM builds.

## Level 2 — FIRST LIGHT *(an OS kernel boots — Layer B)*

**Thesis:** The machine opens its eyes. Level 2 adds exactly the platform Linux needs to
wake up — device tree, SBI firmware interface, a 16550 UART, virtio-mmio transport,
virtio-blk/net, CLINT/PLIC interrupt delivery, timers — and carries an OS kernel from reset
vector to userspace.

**Named milestones:** first **xv6-riscv** (tiny, self-contained — the perfect proof that
Layer A + minimal Layer B is a real machine: UART, virtio-blk, timer), then an **unmodified
Linux kernel** through OpenSBI banner and initramfs to a busybox shell, and on to a full
Alpine riscv64 root filesystem with a `login:` prompt in xterm.js.

**What this level gets you:** *Real Linux in the tab.* Not shims (almostnode), not a mock OS
— the actual unmodified kernel scheduling processes on our CPU. The guest instantly becomes
our most powerful test harness: from here on, correctness questions are answered by running
real software. **Layer G advances here** too: identical device-input sequences must produce
byte-identical boots (deterministic-boot diffs against QEMU are already part of E2's
verification), the first real workout for record/replay.

**Capstone threshold:** In the browser, boot **xv6-riscv** to its shell, then boot an
unmodified Linux kernel from virtio-blk through init; carry Alpine riscv64 to a `login:`
prompt, run `vi`, `top`, shell scripts, and a clean `poweroff`.

## Level 3 — CIVILIZATION *(a real userland — Layer C)*

**Thesis:** A booted kernel is a machine; a *userland* is a computer you can use. Level 3
gives the machine everything a real userspace needs — a persistent rootfs (copy-on-write
block overlay on IndexedDB/OPFS, streaming chunked disk images over HTTP), a network (a
slirp-style user-mode TCP/IP stack in Rust bridging virtio-net primarily through a browser
Tailscale/Headscale node, with a public WebSocket relay as the fallback), working `apk add`
against real mirrors, snapshot/restore, host↔guest transfer — and then proves that real,
dynamically-linked programs run correctly on it: process model, `mmap`/W^X, threads and
futex, signals, TLS. The browser tab is the tailnet node; Alpine sits behind its virtual
gateway unless a later task explicitly gives the guest its own tailnet identity.

**Named milestones:** an interactive **busybox** userland, **QuickJS** (a full embeddable
JS engine — a stress test for libc and mmap), and a **slow, pure-interpreted CLI Node.js**
(the first time a heavyweight real-world runtime executes — correctness before speed). This
is also **feature parity with webvm.io/alpine.html**: a URL that is a persistent, networked
personal Linux box.

**What this level gets you:** Standalone product value, and the correctness foundation for
Level 4. Node.js exercises nearly every hard corner of the userland ABI (futex contention,
`mmap` of large regions, W^X page flips, signal delivery); getting it running *interpreted*
means the machine is correct — Level 4 only has to make it *fast*.

**Capstone threshold:** Cold-load the page, `apk add` the runtimes, drop into a busybox
shell, run a QuickJS program and a Node.js script (interpreted, slow is fine), reach both a
MagicDNS tailnet service and the public internet through the browser node, reload the tab,
and the state and tailnet session are still there — the whole flow comparable to
webvm.io/alpine.html.

## Level 4 — ACCELERATION *(the engine of accelerating returns — Layer D)*

**Thesis:** Kurzweil's curve is powered by each generation of tools building faster tools.
An interpreter at ~50–150 MIPS makes a real Node.js unusable; Level 4 builds a dynamic
binary translator that compiles hot guest code to fresh WASM modules at runtime (the
v86/CheerpX trick), keeps the translated code cache coherent with guest self-modifying code
via **FENCE.I / SFENCE.VMA invalidation** (essential — V8's and JSC's in-guest JITs write
code at runtime), moves execution onto SharedArrayBuffer + workers, and instruments
everything with benchmarks.

**Named milestones:** **interactive, fast Node.js** — the same runtime that crawled at
Level 3, now responsive — and, as a stretch gated on JavaScriptCore's still-immature RISC-V
JIT, **Bun**. This epic is the single highest-leverage one in the whole roadmap:
*performance, not correctness, gates every top target above it.*

**What this level gets you:** 10–50x. The difference between "toy shell" and "you can
develop software inside it." It is deliberately sequenced *after* Level 3: the JIT is
verified by differential execution against the already-trusted interpreter, and **Layer G
must survive the JIT** — record/replay has to reproduce a translated run bit-for-bit, so
the lockstep interpreter-vs-JIT harness doubles as a replay-determinism gate.

**Capstone threshold:** Interactive Node.js in the browser (REPL latency and a real
`npm`-style workload feel usable); ≥10x CoreMark vs. the Level 3 interpreter baseline;
kernel boot < 5 s; zero correctness regressions (full compliance suite passes under JIT).

## Level 5 — THE WINDOW *(graphics, input, sound — Layer E)*

**Thesis:** A civilization builds cities you can see. Level 5 gives the machine a face:
virtio-gpu scanout to canvas/WebGL, virtio-input (keyboard/mouse/tablet) from DOM events,
virtio-snd to WebAudio, and on top of that stack a display server (Wayland via labwc, or
X11) presenting a **rendering surface** for real graphical applications.

**Named milestones:** a working **browser rendering surface** (guest pixels on the canvas,
real keyboard/mouse into the guest) and the first **GUI applications** drawing to it. The
full multi-app *desktop* — especially one mixing x86_64 apps — is Level 7's job; Level 5
proves the surface and the input/output path.

**What this level gets you:** Graphical Linux in a browser tab — beyond alpine.html into
webvm's KMS/DRM territory. This is also where `almostnode`'s almost-os UI work re-enters:
its window-manager shell can become chrome around real guest windows instead of mocks. It is
the prerequisite surface for both Level 7 (a desktop) and Level 8 (Chromium needs somewhere
to paint).

**Capstone threshold:** Boot to a graphical surface; open a GUI application *inside the
guest*; type with your real keyboard; hear a sound; drag a window at usable FPS.

## Level 6 — TRANSCENDENCE *(scale, sharing, self-hosting — Layer G hardened)*

**Thesis:** For the top of the ladder the machine must scale out, become a platform, and see
its *state* turned into a first-class, portable, rewindable object. Level 6 adds SMP (multiple
harts on multiple workers), WebGPU-accelerated 3D (virgl/venus), OPFS/9p shared filesystems,
an embedding SDK (the VM as a component — reconnecting to almostnode's IDE apps), and — the
part that matters most for the summit above — the **Layer G hardening**: versioned/compressed
snapshots, cloud sync, URL-shareable machine states, and the snapshot-as-keyframe substrate
that Level 8's time-travel replay is built on. It closes with the original self-hosting proof:
wasm-vm builds and boots wasm-vm inside itself.

**What this level gets you:** The VM stops being an artifact and becomes a *platform*: any web
page can embed a full Linux machine; machine states are links you can send and *rewind*; the
guest is powerful enough to build the VM itself. SMP and 3D are the horsepower a real desktop
(Level 7) and a full browser (Level 8) demand; the snapshot/replay hardening is what makes
Level 8's "time-travelable" claim mechanically possible.

**Capstone threshold:** Inside the guest: clone this repo, build the core crate, produce a
WASM artifact, serve it, and boot a child VM in an iframe — on an SMP machine, with the
hardened, shareable snapshot machinery Level 8 will scrub through already in place.

## Level 7 — BABEL *(x86_64 inside RISC-V; a real desktop — Layer F)*

**Thesis:** The world's binaries are x86_64. Level 7 teaches the RISC-V machine to speak
them — not with a second CPU, but with **box64** (ptitSeb's x86_64→native userspace
translator) running *as a guest program*, translating x86_64 Linux binaries to riscv64 on
the fly. Around it, Level 7 hardens the two fabrics a daily-driver desktop needs: a **full
network stack** and **robust persistence** at desktop working-set scale. This is
CheerpX's whole value proposition — arbitrary x86 Linux software in the browser — reached
*through* emulation instead of as the base layer.

**Named milestones:** run real **x86_64 Linux binaries** (static, then dynamic; CLI, then
GUI) under box64 inside the machine, and assemble them with riscv64-native apps and the
Level 5 surface into **a real desktop**.

**What this level gets you:** The binary universe. Any x86_64 Linux program becomes runnable
in the tab, composited into a genuine desktop. It is a deliberate *emulation-in-emulation*
bet (a DBT inside our DBT) — Level 4's JIT is what makes box64's translated output fast
enough to be usable, and Level 6's SMP is what makes a heavy desktop bearable.

**Capstone threshold:** Launch an x86_64-only Linux application (verified `file` says
`x86-64`) from the guest desktop; it runs under box64 and renders through virtio-gpu; a
mixed riscv64 + x86_64 desktop session is interactive.

## Level 8 — CHROME IN CHROME *(CANCELLED)*

> **CANCELLED 2026-07-06** (Brett's direction). The nested-browser summit is off the
> roadmap; **Epic 3.5 — OCI Workloads in the Browser** (`tasks/epic-3.5-oci-workloads/`)
> takes the "real software runs in here" crown instead: pull and run genuine riscv64 OCI
> container images, cached in browser storage. The Layer-G record/replay ideas below remain
> archived here and may return as their own epic if VM time-travel becomes a goal again.

**Thesis (archived):** The summit is not our own browser fork — it is **stock Chromium**, unmodified,
booted inside the machine and made *time-travelable by the VM around it*. This is the
inverse of Replay.io's approach: instead of instrumenting Chromium to record itself, we run
an ordinary `chromium-riscv64` build and let the VM's **deterministic record/replay + snapshot
layer** (Layer G, matured across every level below) capture and rewind *the entire machine*.
The determinism lives in the emulator, not in the browser — so the browser can be anything.

**Named milestones:** boot **stock chromium-riscv64** to a rendered page inside the guest
desktop, then wrap the whole run in **record/replay**: capture every source of guest
nondeterminism (device I/O, interrupts, DMA completions, timers, entropy, network) into a
trace, use Level 6 snapshots as keyframes, and **scrub time backward and forward**
deterministically through a live browsing session.

**What this level gets you:** A nested, time-travelable browser — the closed loop of the
whole project. Because the recording surface is the machine and not the application, *no
Chromium-side instrumentation exists*: the exact same stock binary a user would download is
the one that becomes reversible. Past this point the roadmap is open-ended by design —
self-hosting (wasm-vm building and booting wasm-vm inside itself), x86 desktops running
x86 Chrome under box64, distributed disks — because a deterministic, time-travelable Linux
machine on the web defines its own next levels.

**Capstone threshold:** In a fresh browser tab: the VM boots to a desktop, launches stock
chromium-riscv64, loads a real web page and interacts with it; then the operator reverses
execution — snapshot-seek plus deterministic replay-forward — to any earlier point in the
browsing session, bit-identically, with zero Chromium modifications.

## Layer G — determinism & record/replay *(cross-cutting)*

Record/replay is **not a Level-8 bolt-on**; it is a capability that must mature alongside
everything else, because a system that is only made deterministic at the end was never
deterministic. The scaffolding already exists in Level 0 (the trace hook E0-T16, the
snapshot module E0-T17). From there it threads upward:

- **L1 (A):** deterministic instruction execution — identical programs produce identical
  traces, native and WASM (the compliance and native-vs-WASM determinism work *is* Layer G's
  foundation).
- **L2 (B):** deterministic *boot* — identical device-input sequences produce byte-identical
  kernel boots; QEMU-diff verification exercises it.
- **L4 (D):** determinism *under the JIT* — the lockstep interpreter-vs-JIT harness proves a
  translated run replays bit-for-bit; if the JIT isn't deterministic, replay is impossible.
- **L6 (G-hardened):** snapshots become versioned, compressed, integrity-checked *keyframes*;
  shareable/rewindable machine-state URLs; the substrate time-travel is built on.
- **L8 (G capstone):** full record/replay of a live **stock-Chromium** session — capture all
  nondeterminism at the VM boundary, replay bit-identically, scrub time. Determinism in the
  emulator replaces instrumentation in the application.

## How to read the numbers

- Task IDs are `E{level}-T{nn}`; priority = `level × 100 + nn`. The queue is strictly
  ascending priority. Dependencies may pull tasks earlier in *eligibility* but never
  reorder the queue file itself.
- Levels 0–6 carry ~24–28 tasks each; the summit level (7 Babel) is a leaner,
  milestone-focused epic. Level 3.5 (OCI Workloads) is a 5-task wedge between Civilization
  and Acceleration; Level 8 (Chrome in Chrome) is cancelled. Each epic ends in a capstone naming its runnable
  target — you are never guessing what "done" looks like.

*The machine wakes up one instruction at a time — and, at the top, learns to run it backward.*
