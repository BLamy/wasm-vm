# wasm-vm: The Road to the Browser Singularity

**A Kardashev scale for in-browser Linux.**

This document is the guiding trajectory for `wasm-vm`: a from-scratch, Rust-implemented,
WASM-compiled virtual machine that boots real, unmodified Linux in a browser tab — the
end state being full parity with (and then transcendence of) https://webvm.io/alpine.html:
a persistent, networked, GUI-capable Linux system that lives at a URL.

Like Kurzweil's law of accelerating returns, each level here is not just "more features" —
each level is a *phase change* that makes the next level cheaper to reach. The interpreter
makes the CPU debuggable; the compliant CPU makes Linux bootable; booted Linux makes the
system self-testing (the guest becomes our test harness); the JIT makes compilers usable
inside the guest; and at the top, the system compiles itself inside itself. That closed
loop is the singularity condition.

## Prior art & where this project sits

- **webvm.io (CheerpX)** — x86 binary translation to WASM, closed-source engine. The target
  experience to clone.
- **JSLinux / TinyEMU (Bellard)** — proof that RISC-V system emulation boots Linux in JS/WASM.
  The architectural blueprint.
- **v86** — x86 interpreter+JIT in Rust/WASM. Proof the JIT-in-browser approach works.
- **container2wasm** — Bochs/TinyEMU-based container boot in WASM.
- **`~/Dev/almostnode`** — our own prior work: Node.js API shims, virtual FS, mock OS UI
  (`apps/almost-os`), web IDEs. That is the *emulation-of-APIs* approach. `wasm-vm` is the
  *emulation-of-hardware* approach: no shims, no mocks — a machine. The almostnode apps
  become candidate front-ends/shells for this VM at Level 5+.

## The three irreversible architectural bets

1. **Guest ISA: RISC-V (RV64GC), not x86.** CheerpX spent years on x86 translation. RV64GC
   is a clean, fully-open, formally specified ISA with an official compliance suite
   (riscv-tests, RISCOF), and Alpine Linux ships riscv64 as a release architecture. Every
   binary webvm runs has an Alpine riscv64 equivalent. This single decision cuts the CPU
   effort by ~10x. (An x86 translation layer is a possible post-Level-6 moonshot, not a goal.)
2. **Language/runtime: Rust → `wasm32`, no_std-friendly core.** The emulator core is a pure
   Rust crate with zero web dependencies (testable natively at native speed), wrapped by a
   thin `wasm-bindgen` boundary. Everything browser-specific (IndexedDB, WebSocket, canvas,
   WebAudio, workers) lives behind Rust traits so the core never knows it's in a browser.
3. **Device model: virtio everywhere.** Block, network, GPU, input, sound, filesystem — all
   virtio-mmio. One transport, one ring-buffer implementation, amortized across every device.
   The Linux kernel already has mature drivers for all of them; we never write a guest driver.

## Execution doctrine

- All work is decomposed into tasks under `tasks/`, organized into the seven Epics below.
- A **priority queue** (`tasks/QUEUE.md`, regenerated by `tools/build_queue.py`) totally
  orders all tasks. Work proceeds one task at a time, top of queue first.
- Every task ends with **adversarial verification**: a verifier whose explicit mission is to
  *refute* the task's completion claim (see `tasks/README.md` for the protocol). A task is
  done when a hostile verifier fails to break it, not when its implementer says it works.
- Each Epic ends in a **capstone task** — a demonstrable, user-visible threshold. You cannot
  start Epic N+1's capstone-dependent work until Epic N's capstone survives adversarial
  verification.

# The Scale

```
Level 0  IGNITION        a Rust machine skeleton executing RISC-V in a browser tab
Level 1  THE MACHINE     a spec-compliant RV64GC CPU with MMU, traps, timers, interrupts
Level 2  FIRST LIGHT     unmodified Linux boots to a login prompt
Level 3  CIVILIZATION    persistent, networked, package-installing Alpine — webvm parity
Level 4  ACCELERATION    JIT to WASM; 10–50x; compilers become usable inside the guest
Level 5  THE WINDOW      virtio-gpu + input + sound; a real Linux desktop in the tab
Level 6  TRANSCENDENCE   SMP, WebGPU 3D, shareable machine states, self-hosting
```

---

## Level 0 — IGNITION *(the substrate)*

**Thesis:** Before a machine can think, it must have a body. Level 0 builds the skeleton:
the workspace, the memory bus, a minimal RV64I interpreter, the trace/debug organs, the
native + browser test harnesses, and the adversarial-verification machinery itself.

**What this level gets you:** A Rust core crate that executes bare-metal RISC-V test
binaries identically under `cargo test` (native) and in a browser tab (WASM), with an
instruction-level trace you can diff against a reference emulator (QEMU/Spike). From this
point forward, every bug is *observable* — nothing later is built on an undebuggable base.

**Capstone threshold:** A browser page loads the WASM module, executes a compiled bare-metal
RISC-V program that prints "Hello from RV64" through a stub console, and the instruction
trace matches Spike's, byte for byte.

## Level 1 — THE MACHINE *(a complete CPU)*

**Thesis:** Compliance is non-negotiable. Linux does not boot on a CPU that is 98% correct.
Level 1 turns the interpreter into a full RV64GC machine: I/M/A/F/D/C extensions, M/S/U
privilege modes, CSRs, precise exceptions and interrupts, the Sv39 MMU with TLB, CLINT
(timers/IPIs) and PLIC (external interrupts).

**What this level gets you:** A CPU that passes the official riscv-tests and RISCOF
architectural compliance suites. This is the difference between "an emulator that runs my
demos" and "hardware." After Level 1, when Linux misbehaves, the bug is in our devices or
firmware — never silently in the CPU. That certainty is what makes Level 2 a finite task.

**Capstone threshold:** Green across riscv-tests (rv64ui/um/ua/uf/ud/uc, mi/si) and a RISCOF
run, in both native and WASM builds.

## Level 2 — FIRST LIGHT *(Linux boots)*

**Thesis:** The machine opens its eyes. Level 2 adds exactly the devices Linux needs to
wake up — device tree, SBI firmware interface, a 16550 UART, virtio-mmio transport,
virtio-blk — and carries the kernel from `OpenSBI` banner through initramfs/busybox to a
full Alpine riscv64 root filesystem and a `login:` prompt in xterm.js.

**What this level gets you:** *Real Linux in the tab.* Not shims (almostnode), not a mock OS
— the actual unmodified kernel scheduling processes on our CPU. The guest instantly becomes
our most powerful test harness: from here on, correctness questions are answered by running
real software. This is the level where the project becomes demoable to anyone.

**Capstone threshold:** Boot unmodified Alpine riscv64 from a virtio-blk ext4 image to a
login shell in the browser; run `vi`, `top`, `sh` scripts; clean `poweroff`.

## Level 3 — CIVILIZATION *(a usable system — webvm parity)*

**Thesis:** A machine that forgets everything on reload is a demo; a machine with memory
and communication is a civilization. Level 3 adds persistence (copy-on-write block overlay
on IndexedDB/OPFS, streaming chunked disk images from HTTP), networking (a slirp-style
user-mode TCP/IP stack in Rust bridging virtio-net to fetch/WebSocket/WebTransport, plus a
Tailscale-style relay option), `apk add` working against real mirrors, snapshot/restore,
host↔guest file transfer, and terminal/UX polish.

**What this level gets you:** **Feature parity with webvm.io/alpine.html.** A URL that is a
persistent personal Linux box: your files survive reload, you install packages, you `curl`
the internet, you share disk state. This is the level with standalone product value — every
level after this is about *how much* machine, not *whether* there is one.

**Capstone threshold:** Cold-load the page, `apk add python3`, write a script, reload the
tab, and the script is still there and runs — with the whole flow feeling comparable to
webvm.io.

## Level 4 — ACCELERATION *(the engine of accelerating returns)*

**Thesis:** Kurzweil's curve is powered by each generation of tools building faster tools.
An interpreter at ~50–150 MIPS makes big compiles painful; Level 4 builds a dynamic binary
translator that compiles hot guest code to fresh WASM modules at runtime (the v86/CheerpX
trick), moves execution off the main thread onto SharedArrayBuffer + workers, and
instruments everything with benchmarks (CoreMark, Dhrystone, real `gcc`/`rustc` timings).

**What this level gets you:** 10–50x. The difference between "toy shell" and "you can
develop software inside it." Compilers, interpreters, and package builds become interactive.
This is the enabling level for both the desktop (Level 5 needs frames per second) and
self-hosting (Level 6 needs to run rustc). It is deliberately sequenced *after* Linux works:
the JIT is verified by differential execution against the already-trusted interpreter.

**Capstone threshold:** ≥10x CoreMark vs. the Level 3 interpreter baseline; kernel boot
< 5 s; `gcc -O2` of a non-trivial C file inside the guest at interactive speed; zero
correctness regressions (full compliance suite passes under JIT).

## Level 5 — THE WINDOW *(graphics, input, sound — a desktop)*

**Thesis:** A civilization builds cities you can see. Level 5 gives the machine a face:
virtio-gpu scanout to canvas/WebGL, virtio-input (keyboard/mouse/tablet) from DOM events,
virtio-snd to WebAudio, and on top of that stack a real display server (Wayland via labwc,
or X11) running a lightweight desktop with working clipboard, cursor, and resize.

**What this level gets you:** A full graphical Linux desktop in a browser tab — beyond
alpine.html into webvm's KMS/DRM demos. Firefox-in-a-tab-in-a-tab territory. This is also
where `almostnode`'s almost-os UI work re-enters: its window-manager shell can become a
chrome around real guest windows instead of mocks.

**Capstone threshold:** Boot to a graphical desktop; open a terminal emulator *inside the
guest desktop*; type with your real keyboard; hear a sound; drag a window at usable FPS.

## Level 6 — TRANSCENDENCE *(the self-hosting singularity)*

**Thesis:** Kurzweil's singularity is the point where the system improves itself. For an
in-browser VM the closed loop is concrete and testable: **wasm-vm compiles wasm-vm, inside
wasm-vm, and boots the child.** Level 6 adds SMP (multiple harts on multiple workers),
WebGPU-accelerated 3D (virgl/venus), OPFS/9p shared filesystems, URL-shareable and
cloud-syncable machine snapshots, an embedding SDK (the VM as a component — reconnecting to
almostnode's IDE apps), and finally the self-host capstone.

**What this level gets you:** The VM stops being an artifact and becomes a *platform*: any
web page can embed a full Linux machine; machine states are links you can send; the guest is
powerful enough to develop, build, and run the VM itself. Past this point the roadmap is
open-ended by design — x86 translation layers, KVM-like nested acceleration, distributed
disks — because a self-hosting system defines its own next levels.

**Capstone threshold:** Inside the guest: clone this repo, build the core crate, produce a
WASM artifact, serve it, and boot a child VM in an iframe — grandchild optional but
glorious.

## How to read the numbers

- Task IDs are `E{level}-T{nn}`; priority = `level × 100 + nn`. The queue is strictly
  ascending priority. Dependencies may pull tasks earlier in *eligibility* but never
  reorder the queue file itself.
- Each epic contains 22–28 tasks. Expect ~170 tasks total. At one verified task per
  working session, Level 3 (webvm parity) is ~100 sessions out; the curve after that
  accelerates — see thesis, Level 4.

*The machine wakes up one instruction at a time.*
