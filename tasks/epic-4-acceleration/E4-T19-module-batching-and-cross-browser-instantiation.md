## Goal
The one-Module-per-block placeholder becomes a measured batching strategy: hot blocks are
compiled K-per-module (translation-group batching), with K and the compile pathway
(async `WebAssembly.compile` vs `compileStreaming`, main-thread vs worker-side) chosen
from a scripted cost matrix run on Chrome, Firefox, and Safari — because per-Module fixed
overheads (compile setup, tiering, per-Instance metadata) differ wildly across engines and
browsers throttle or cap pathological module counts.

## Context
v86 batches many basic blocks into each generated module and keeps module count low; naive
one-block-one-module designs drown in per-module overhead and browser instance limits.
Batching also buys free intra-module chaining: blocks in one module call each other with
direct `call` (no table indirection), so batch composition interacts with E4-T18 (group
blocks by observed edges — batch the current compile queue's connected components).
Measurables per browser: compile latency vs module size (1..256 functions), instantiate
latency, memory per Module+Instance (via performance.memory / about:memory where
available), behavior at 1k/5k/10k live Instances, and warm-up (does the engine tier up
JIT-emitted modules — Liftoff→TurboFan etc. — and when). Safari runs via WebKit GTK/
Playwright if no macOS CI runner exists; document what was actually measured where.

## Deliverables
- `bench/module-costs/`: scripted harness (Playwright matrix) emitting per-browser JSON:
  compile/instantiate latency curves, memory per module, instance-count cliff tests.
- Batching implementation: compile queue drained in groups; intra-batch direct calls for
  known edges; per-batch funcref-table export for cross-batch links (E4-T18 slots).
- Chosen K + pathway per browser (UA-probed defaults, one override knob), justified by
  the measurements in `docs/jit-architecture.md` amendment.
- Instance registry: every live Module/Instance accounted (count + estimated bytes) —
  the raw material for E4-T20's budgets.

## Acceptance criteria
- [ ] Cost-matrix JSON committed for Chrome + Firefox (+ Safari or documented substitute),
      generated by one command from a clean checkout.
- [ ] Batched vs unbatched A/B on the gcc benchmark (compile-heavy = translation-heavy):
      batching reduces total JIT-compile stall time by a measured, recorded factor.
- [ ] Intra-batch direct calls verified in emitted bytes (no call_indirect for
      within-batch edges — asserted by a codegen test).
- [ ] The instance-count cliff test documents each browser's practical limit and the
      chosen budget stays ≥ 4x below it.
- [ ] Full riscv-tests green with batching on (block↔batch mapping introduced no lookup
      bugs).

## Adversarial verification
Refute the measurements' relevance and the batching's correctness. Attack angles:
(1) rerun the cost matrix on a different machine/browser-version — if the chosen K flips
to a >2x-worse choice, the "measured decision" is overfit and refuted as stated (the task
must show K is robust or auto-tuned); (2) partial-batch invalidation: SMC-kill one block
out of a 64-block module and verify the other 63 still execute, the dead one is unlinked
(intra-module edges too — these are direct calls, so the dead block's *body* must be
neutered via its prologue generation check or the whole batch retired — whichever the
design says, test it); a stale intra-batch direct call executing dead code refutes;
(3) instance-limit abuse: force-tune K=1 and run gcc until 10k modules exist — the VM must
hit its own registry accounting (E4-T20 will enforce; here, accounting must at least be
accurate within 10% of browser-reported memory growth); (4) measure first-execution
latency of a fresh batch (engine baseline-compile warm-up) and confirm it doesn't violate
the E4-T06 pause targets on the worst browser.
