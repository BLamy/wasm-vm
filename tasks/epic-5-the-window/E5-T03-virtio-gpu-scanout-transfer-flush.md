## Goal
The device implements the three commands that actually move pixels: `TRANSFER_TO_HOST_2D`
copies a rect from guest backing pages into the host shadow buffer, `SET_SCANOUT` binds a
resource to scanout 0, and `RESOURCE_FLUSH` hands a damage rect to a host-side `FrameSink`
trait — testable natively with zero browser code.

## Context
This is the display's data plane. TRANSFER_TO_HOST_2D carries
`{rect{x,y,w,h}, le64 offset, resource_id}`; the source is the backing sglist treated as
one linear byte stream (stride = resource width * 4, offset addresses into it), the
destination is the host shadow at the same rect. SET_SCANOUT with `resource_id 0` disables
the scanout (guest blanking — fbcon does this). RESOURCE_FLUSH publishes the shadow's rect
to whatever presents it; per bet #2 the core crate must not know about canvas, so it calls
`FrameSink::flush(scanout, rect, &[u32])` behind a trait. BGRA vs RGBA swizzle for the
browser happens at the sink boundary, not in the core. Reference: virtio v1.2 §5.7.6.8;
QEMU virtio-gpu 2D path.

## Deliverables
- Handlers for the three commands with full rect validation (rect within resource dims,
  offset+span within backing size, scanout_id < num_scanouts).
- Scatter-gather copy that handles rects whose rows cross sg-entry boundaries.
- `FrameSink` trait in the core crate; `TestSink` recording (rect, CRC32 of pixels) for
  native tests; a null sink for headless boots.
- Flush on a non-scanout-bound resource: legal, forwarded with resource dims (guest
  cursors and fbcon use this pattern before binding).
- Golden-image test: render a known test pattern via guest-page writes + TRANSFER + FLUSH
  and compare CRC against a fixture generated by an independent Rust reference path.

## Acceptance criteria
- [ ] Full-frame and partial-rect transfers produce CRC-identical output to the reference
      for 5 fixture patterns, including a rect starting at odd x with w not a multiple
      of the sg-entry size.
- [ ] TRANSFER with rect exceeding resource bounds returns `ERR_INVALID_PARAMETER` and
      writes nothing to the shadow (shadow CRC unchanged).
- [ ] SET_SCANOUT with resource_id 0 disables output; subsequent FLUSH of the old
      resource still succeeds but the sink sees no scanout-bound flush.
- [ ] TRANSFER on a resource with detached backing errors without reading guest memory.
- [ ] Same test results native and wasm32.

## Adversarial verification
Refute via boundary arithmetic: fuzz 10k random (rect, offset) pairs against a
model-based oracle (naive nested loop copy) and diff shadow CRCs — any divergence
refutes. Attack the sg walk: craft a backing whose entries are 1 byte each and transfer a
64x64 rect; craft offset = backing_len - 1 with a 2-byte span (must error). Prove no
guest-memory *write* ever occurs on this path (transfers are read-only from guest;
instrument the bus). Time a 1280x800 full-frame transfer natively; > 5 ms indicates a
per-pixel sg lookup (algorithmic refutation — must be per-run, not per-pixel).
