## Goal
The unavoidable "get the image in" step between `oci unpack` (E3.5-T01) and running a real
container (`wvrun`, E3.5-T03) toward the `wvrun postgres` capstone: fetch a public OCI image for
a target arch into a local OCI image-layout that `wasm-vm oci unpack` already consumes — WITHOUT a
new Rust HTTP dependency or a registry client baked into the emulator. The digest-deduped BROWSER
layer cache is the separate E3.5-T04; this is the native dev-side pull.

## Context
`postgres:latest` publishes a real **riscv64** image on Docker Hub (verified 2026-07-06), so the
beeline is feasible. Rather than add `ureq`/`reqwest` + registry auth to the CLI (an architectural
commitment whose test can't be CI-deterministic — it hits the network), the pull is a small shell
tool (`tools/oci-sideload.sh`) over `curl` + `python3` (already required by the build tooling),
using Docker Hub anonymous pull tokens: resolve index → arch manifest → fetch config + every layer
into `blobs/sha256/<hex>`, **sha256-verifying every blob as it lands**, and emit `index.json`. The
output is byte-for-byte the layout `oci unpack` resolves. Re-runs skip already-verifying blobs
(poor-man's cache). A native Rust `oci pull` (if wanted later) is a clean follow-up; a
`skopeo copy oci:dir` layout also drops straight in.

## Deliverables
- `tools/oci-sideload.sh <repo>[:<tag>] <out-dir> [<arch>]` → an OCI image-layout, every blob
  digest-verified, idempotent re-runs.
- `oci unpack` robustness fix for real tarballs (below) + a deterministic regression test.

## Acceptance criteria
- [x] Sideload `busybox` riscv64 → `oci unpack` → a runnable bundle: 447 entries, `run.json`
      argv=`["sh"]` + the image's real PATH, `rootfs/bin/sh` is a **RISC-V 64-bit ELF** (verified
      end-to-end natively, 2026-07-06).
- [x] A blob served with wrong bytes fails the sideload with a digest mismatch (the script
      re-hashes every blob; deletes + exits non-zero).
- [ ] Sideload `postgres` riscv64 → unpack → bundle (larger, multi-layer; the capstone input) —
      pending (bigger download; the mechanism is proven on busybox).

## Adversarial verification
The registry is untrusted: every blob (manifest, config, layer) is sha256-checked against its
requested digest before use — wrong-bytes-under-a-correct-key is caught by the script AND again by
`oci unpack`'s `read_verified`. A truncated/partial download fails the hash. The arch picker must
select the requested architecture from the manifest list (proven: busybox came back RISC-V ELF, not
host amd64). Cold-clone critic on this diff: pending.

## Verification log

**2026-07-06 — native sideload + a REAL-image bug fix (pass 1).** Confirmed `postgres:latest` has a
riscv64 image; verified the pull mechanism (index→riscv64 manifest→config+layers, digest-checked).
`tools/oci-sideload.sh` pulls a repo:tag into an OCI layout over curl+python3, no new Rust dep.
**Sideloading the real busybox riscv64 image caught a genuine `oci unpack` bug my synthetic tests
missed:** every real image tarball carries a `./` (and often `.`) root-directory member, which
normalizes to the empty path and `safe_path` (correctly) rejects — so unpack FAILED with
`UnsafePath("./")`. Fix: the tar reader now SKIPS the root entry (`.`/`./`) before it reaches the
applier, while `..`/absolute paths stay rejected. Deterministic regression test
`tar_root_dir_entry_is_skipped_not_rejected` added (10/10 oci tests green). After the fix, the full
chain works end-to-end natively: sideload busybox → `oci unpack` → 447-entry bundle, correct
argv/env/cwd from the real config, `rootfs/bin/sh` = riscv64 ELF. Next: postgres sideload +
`wvrun` boot proof (long-boot, deferred — env kills long runs).

**2026-07-06 — adversarial cold-clone critic (FIX-FIRST; fixes applied + re-proved).** Critic ran a
live sideload + tamper tests. One MAJOR (confirmed by repro) + MINORs, all fixed:
- **MAJOR — manifest fetched by digest was NOT verified against that digest.** `fetch_manifest_blob`
  hashed whatever bytes came back and content-named the file, so the index→manifest integrity link
  was never enforced — a corrupted/lying manifest was accepted and it dictates which config/layer
  digests get pulled. (config + every layer WERE verified; the manifest was the gap.) FIX: when the
  ref is a `sha256:` digest (the arch manifest picked from the index), verify the received bytes hash
  to it, else abort exit 3. Re-proved: mismatch → REJECTED.
- **MINOR — single pull token, no refresh.** Docker Hub tokens expire ~300s; the multi-layer
  postgres pull (the actual goal) could 401 mid-pull. FIX: `authed_get` helper refreshes the token
  once on a 401 and checks the HTTP code (so an error page can't masquerade as a blob).
- **LOW — function locals leaked to globals** (`ref`/`hex`/`dst`/…): added `local`.
- **LOW — silent misparse of `repo@sha256:` digest-pins and registry-host prefixes**: now rejected
  with a clear message (Docker-Hub-tag-only tool). Re-proved: both error cleanly.
Critic CONFIRMED SOUND: the `oci unpack` root-skip fix opens NO containment hole (skip fires only for
literal root `.`/`./`/empty; `..`/absolute still hit `safe_path` and are rejected — 10/10 oci + 10/10
storage-oci tests incl. all escape tests); config+layer digest verification correct; arch picker
exact-match (skips `unknown` attestation manifests, errors on missing arch); no shell injection from
`ref`. `bash -n` clean; busybox end-to-end still green after fixes.
