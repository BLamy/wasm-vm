## Goal
Pull an OCI image (manifest list → platform manifest → config + layer blobs) for
`linux/riscv64` from a registry, verify every content digest, and unpack the layer stack
(whiteout-aware) into a guest-visible rootfs — either written into the CoW overlay as a
directory tree the guest can chroot/pivot into, or staged in a browser-backed layer cache
(E3.5-T04) that materializes on demand.

## Context
The pull happens HOST-SIDE (browser `fetch` / native reqwest-less HTTP like E3-T02's layer),
NOT from inside the guest — no guest networking dependency, and the E3-T02 fetch
infrastructure (request/verify/retry protocol logic in `crates/storage`, browser transport
in `crates/wasm`) is the pattern to extend. Registry protocol: OCI Distribution Spec pulls
(`/v2/<name>/manifests/<ref>` with the OCI+Docker media types, `/v2/<name>/blobs/<digest>`),
token auth for public registries (anonymous pull token from `auth.docker.io`-style
endpoints), CORS reality check early — ghcr.io/Docker Hub may need the E3-T16/T19 proxy for
browser pulls; a native CLI pull path (`wasm-vm oci pull`) keeps the format/unpack logic
testable without a browser. Digest verification is NON-OPTIONAL: sha256 of every blob
checked against the manifest before unpack (reuse the E3-T01 hashing). Layer unpack:
gzip/zstd tar, applied in order with OCI whiteout semantics (`.wh.` files, opaque dirs).
Keep the tar/whiteout applier in `crates/storage` (no_std-friendly where possible, std for
the CLI) so the browser and native paths share it. riscv64 images exist for alpine, busybox,
and most official images — pick alpine:latest (linux/riscv64) as the reference.

## Deliverables
- Registry client (manifest resolution incl. multi-arch index → riscv64, token auth, blob
  fetch with digest verify) usable from native CLI; browser transport variant.
- Tar + whiteout layer applier producing a flattened rootfs tree from an ordered layer set.
- `wasm-vm oci pull <image>` CLI: pulls to a local content-addressed store
  (`~/.wasm-vm/oci/blobs/sha256/...` layout mirroring OCI image-layout) + `wasm-vm oci
  unpack <image> <dir>`.
- A path from unpacked rootfs → guest-visible filesystem: inject into the ext4 image build
  (tools/build-rootfs.sh extension) or write into the overlay at a mount point the guest
  can reach (document the chosen mechanism; loop-mount of a generated image is fine v1).
- Native tests: golden small image (vendored fixture blobs, no network) → unpack →
  byte-exact expected tree incl. whiteout cases; digest-mismatch refusal.

## Acceptance criteria
- [ ] `wasm-vm oci pull alpine:latest` (linux/riscv64) resolves the index, fetches config +
      all layers, verifies every sha256, and stores them content-addressed; a second pull
      is a no-op (cache hit by digest).
- [ ] A corrupted blob (single flipped byte) is refused with a typed digest-mismatch error
      naming the expected/actual digest — never unpacked.
- [ ] Whiteout semantics proven: a fixture with `.wh.file`, an opaque dir, and a
      file-replaces-dir case unpacks to the documented expected tree.
- [ ] The unpacked alpine rootfs boots as a guest filesystem (chroot or root=) far enough
      to run `/bin/busybox true` (exit 0) natively.
- [ ] Browser leg: the importer fetches + verifies a small image's blobs in-browser
      (CORS-viable registry or the dev proxy) — measured and logged.

## Adversarial verification
Hostile registry: truncated blobs, wrong media types, a manifest whose layer list omits a
digest the config references, gzip bombs (cap decompressed size), tar entries escaping the
root (`../`, absolute paths, symlink-then-write-through) — any file materializing outside
the target rootfs refutes. Digest checks must happen on the COMPRESSED blob bytes (the
descriptor digest) — verify against a registry-pulled fixture, not just self-made ones.
