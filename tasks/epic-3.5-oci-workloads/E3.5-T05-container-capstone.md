## Goal
The full loop in a clean browser: boot Alpine, pull a small riscv64 OCI image from a real
registry, run it with the tiny runner and see its output in the terminal; reload the tab,
run it again — this time from the persisted layer cache with zero blob downloads. Then, as
the graduation exercise (gated on guest port-forwarding), a Node/HTTP container serving a
request that a host-browser fetch can hit.

## Context
This is the "civilization ships containers" moment and the proof that E3.5-T01→T04 compose:
importer (pull+verify+unpack) → persistence (cache, reload survival) → runner (namespaces,
overlay, stdio) → all inside the browser-hosted VM. Playwright drives it exactly like the
E3-T05 double-boot test (fresh profile = clean IndexedDB; request interception counts blob
fetches). Image choice: the smallest honest one that proves execution (busybox or
alpine-based, riscv64) — output must be computed in-container (echo-proof marker
discipline: `CONTAINED_$((6*7))`, never a literal the transcript already carries). The
**graduation leg** — a Node (or busybox httpd) container answering HTTP from the host
browser — needs guest→host port-forwarding, which is E6-T25 (or the E3-T16 proxy path run
in reverse); it is an EXPLICIT second checkbox gated on that dependency landing, not a
silent scope cut: record it as deferred-with-pointer if this capstone verifies first.

## Adversarial verification
Run the capstone in a genuinely fresh profile (no warm state) — a stale-profile pass
refutes nothing. Interception must count ALL network bytes, not just XHR (fetch + workers).
Prove the second run is the cache and not the registry: run it offline (context.setOffline)
— it must still work. Tamper one cached layer before the second run — it must refetch (or
fail typed offline), never run corrupted code. Diff the in-container marker computation
against a literal-echo trap (the E3-T13 F1 lesson: transcript echo makes `contains()`
vacuous — markers must be computed in-guest).

## Verification log

**2026-07-07 — pass 1: real-runner groundwork (`build-container-bundle.sh`) + CRITICAL importer fix
(hardlink preservation).** Building the real path (busybox-first) for the demo's Docker tab — replacing
the fake in-page JS shell — immediately surfaced a real bug the mock hid: `wasm-vm oci unpack` was
EXPANDING tar hardlinks into full COPIES, so busybox (whose ~410 applets are all hardlinks to one
binary) unpacked to a **412 MiB** rootfs instead of ~3 MiB (~130× bloat) — infeasible to bake/ship.
Root cause: the `Entry::Hardlink` apply cloned the target `Node::File`'s bytes; there was no hardlink
node. Fix: a real `Node::Hardlink { target }` (chains collapsed to the ultimate file at apply), and
`write_tree` creates ACTUAL `std::fs::hard_link`s in a SECOND pass (the sorted tree can place a link,
e.g. `bin/[`, before its target `bin/busybox`). **Result verified natively: busybox rootfs 412 MiB →
3.2 MiB, `bin/[` and `bin/sh` share one inode, `sh` still a RV64GC RISC-V ELF.** New tool
`tools/build-container-bundle.sh <ref> <out> [arch]` wraps sideload+unpack and ASSERTS the entry binary
is an ELF for the target arch (no fake/wrong-arch bundle passes) + emits a manifest.json with REAL
metadata (image ref, manifest digest, entry argv, on-disk size via `du` — link-correct). Proven here:
`busybox riscv64` → real 448-entry, 3.2 MiB bundle, entry `sh` = RISC-V ELF. Tests: storage
`symlink_and_hardlink_apply` updated to assert `Node::Hardlink` (was asserting the buggy copy) +
`hardlink_chain_collapses_to_the_ultimate_file`; cli `write_tree_preserves_hardlinks_as_real_on_disk_links`
(shared inode, two-pass order). 28 storage + cli oci tests green; fmt clean. NOTE: this is groundwork —
the browser wiring (gut `docker.js`'s mock; boot the real `WasmLinux` + run `wvrun` + attach the real

**Adversarial cold-clone critic on pass 1: MAJOR found in the FIRST hardlink fix — re-fixed correctly.**
The initial fix modeled a hardlink as a live PATH reference (`Node::Hardlink { target }`). The critic
proved (repro tests) this REGRESSED correctness two ways when a later layer touches the target: (1)
**whiteout of the target → the link dangles → the WHOLE `oci unpack` aborts** (`hard_link` ENOENT);
(2) **override of the target → the link silently captures the NEW content** (overlayfs copy-up gives
the new file a fresh inode, so the link should keep the OLD content). Both are overlayfs-legal in
multi-layer images. Root cause: a path reference into the mutable tree is invalidated by later
mutations. Correct re-fix: keep **copy-at-apply** semantics (a hardlink resolves to the target's
CONTENT as its own node — immune to later whiteout/override, the semantically-correct behavior the
ORIGINAL code had) and move the size optimization to WRITE time: `write_tree` de-dups identical
`(mode, sha256(bytes))` into real `std::fs::hard_link`s (first path written, rest linked; falls back
to a copy if the fs refuses a link, so unpack never fails on the optimization). This fixes BOTH
MAJORs AND keeps busybox small: **rootfs 3.2 MiB, `bin/[`/`bin/sh` one inode, nlink 411, `sh` a RISC-V
ELF.** Critic-recommended regressions added: `hardlink_survives_whiteout_of_its_target` (b keeps v1,
no dangle/abort) + `hardlink_keeps_old_content_when_target_is_overridden` (a=v2, b=v1) at the apply
level, and `write_tree_dedups_identical_content_into_one_inode` (5 applets → one inode, distinct
content stays separate) at the write level. The critic also verified SOUND: no security regression
(hardlink-to-symlink/dir falls back to node-clone through the guarded write path; `safe_path` + the
apply-time and write-time symlink-ancestor guards all hold). 28 storage+cli oci tests green; fmt +
clippy clean.
