## Goal
Close the generality gap the container matrix (E3.5-T04b) surfaced: `oci unpack` decompressed
**gzip only**, so any image whose layers use **zstd** (modern buildkit / `docker buildx
--compression=zstd`, and the OCI `…tar+zstd` media type) was mis-read as a raw tar and failed. To
"run any riscv-compatible container" the importer must handle zstd layers transparently.

## Context
The layer decompressor in `crates/cli/src/oci.rs` (`apply_layer_tar_capped`) sniffs the magic:
gzip `1f 8b`, else stored tar. Added zstd `28 b5 2f fd` → decode via the bundled-libzstd `zstd`
crate (decoder only; battle-tested against hostile input, which matters for untrusted registry
blobs). Crucially it plugs into the SAME streaming path — `Read` → `Read::take(budget+1)` →
`CountingRead` — so the existing decompression-bomb cap (E3.5-T01 critic MAJOR) guards zstd bombs
too, for free. `cli` is native-only (not compiled to wasm), so the C libzstd build is fine in CI.

## Deliverables
- zstd detection + streaming decode in `apply_layer_tar_capped`, bounded by the same budget cap.
- Tests: a zstd layer unpacks identically to gzip; a zstd bomb hits the cap (not buffered).
- End-to-end: a real image layer recompressed as zstd unpacks through the CLI to the correct arch.

## Acceptance criteria
- [x] A zstd-compressed layer (`28 b5 2f fd`) unpacks to the same tree as its gzip equivalent
      (`zstd_compressed_layer_unpacks`).
- [x] A zstd decompression bomb (64 MiB→KB) is refused by the streaming cap, not buffered
      (`zstd_bomb_is_capped`).
- [x] End-to-end: busybox's real layer recompressed to `tar+zstd`, wrapped in an OCI layout with the
      `application/vnd.oci.image.layer.v1.tar+zstd` media type, unpacks via `wasm-vm oci unpack` →
      447 entries, `rootfs/bin/busybox` = RISC-V 64-bit ELF (verified 2026-07-06).
- [ ] (Nice-to-have) a public riscv64 image that actually ships zstd layers unpacks — none found in
      the T04b spread (all gzip); the synthetic + recompressed-real proofs stand in.

## Adversarial verification
The zstd path must be bounded exactly like gzip: a bomb must hit the cap, and a truncated/garbage
zstd stream must be a typed error, never a panic or unbounded read. Confirm the same
`CountingRead`/`take(budget+1)` wraps the zstd decoder (a decoder that reads ahead internally must
still not exceed the budget in delivered bytes). Confirm digest verification still runs on the
compressed blob before decompression (a zstd blob is verified by `read_verified` like any other).

**2026-07-06 — adversarial cold-clone critic: SOUND, ship it (no FIX-FIRST).** The critic attacked
the two high-value properties and could not break either, confirming by repro against real hostile
zstd blobs:
- **Bomb bound holds.** Construction order verified: `Decoder` → `take(budget+1)` → `CountingRead`
  (cap on decompressed OUTPUT, never the compressed input). A **10 GiB logical bomb** (327 KB blob)
  capped in **1.5 ms** — the streaming `read::Decoder` does NOT preallocate to the frame's declared
  size (unlike `decode_all`). A forced **large-window frame** (`windowLog 31`) is REFUSED by
  libzstd's `windowLogMax` (~128 MiB), not allocated — typed error, no OOM.
- **Panic-free on hostile input:** 8 inputs (truncated / garbage / zstd-of-non-tar / zstd-of-corrupt-
  tar / empty / magic-then-EOF / random-window / skippable-frame-prefix) → all typed `UnpackError`,
  zero panics (wrapped in `catch_unwind`).
- **Digest-before-decode** confirmed (`read_verified` runs before the decoder); **magic-sniffed not
  label-trusted**; **wasm job safe** (cli is a native `[[bin]]`, libzstd compiles only on native).
Adopted the two cheap LOW notes: (LOW-1) a comment documenting the ~128 MiB zstd window memory floor;
(LOW-3) strengthened `zstd_bomb_is_capped` to a **512 MiB-logical** bomb built by STREAMING zeros
through the encoder (plaintext never materialized; asserts the blob is <0.1% of logical) so a future
swap to a buffering `decode_all` would regress loudly. LOW-2 (skippable-frame-prefix misclassified)
left as a noted theoretical generality gap — it errors cleanly, and real layer blobs never do it.
