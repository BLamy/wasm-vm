## Goal
An audited, documented kernel configuration for running containers in the guest: confirm
(and where missing, enable + rebuild) namespaces, cgroups, seccomp, overlayfs, veth/bridge/
NAT, tmpfs, and loop devices — with an in-guest proof for each (not just a config grep).

## Context
**Audited 2026-07-06 against the shipped `releases/kernel/6.6.63/config`** (the E3-T13
network-enabled rebuild pulled most of this in via the riscv defconfig):
ALREADY ON — `NAMESPACES` + `PID/NET/USER/IPC/UTS_NS`, `CGROUPS` + `MEMCG/CPUSETS/
CGROUP_PIDS/CGROUP_CPUACCT`, `SECCOMP` + `SECCOMP_FILTER`, `OVERLAY_FS`, `VETH`, `BRIDGE`,
`NETFILTER` + `NF_NAT`/`IP_NF_NAT`, `TMPFS`, `BLK_DEV_LOOP`.
CURRENTLY OFF — `TUN` (needed for some container network modes; not for veth/bridge v1)
and `SQUASHFS` (only if we ship squashed layers). Decide each explicitly: enable in
`configs/wasm-vm.config` + rebuild via `tools/build-kernel.sh`, or document why deferred.
Also verify the runtime side: cgroup v2 mount works (`/sys/fs/cgroup`), `unshare`/`nsenter`
exist in the guest userland (busybox has unshare; Alpine's util-linux if not), and
`pivot_root` into an overlayfs-backed rootfs works on our emulated MMU — an emulator bug
in any syscall here (clone3 flags, setns, pivot_root, mount) surfaces NOW rather than
mid-runner (E3.5-T03).

## Deliverables
- `docs/containers.md`: the audited config matrix (symbol → state → why), the TUN/SQUASHFS
  decisions, and the in-guest proof transcript for each capability.
- Any `configs/wasm-vm.config` additions + rebuilt pinned kernel (if a symbol is enabled).
- An in-guest smoke script (`tools/guest/container-smoke.sh`, shipped into the rootfs):
  unshare each namespace type, write a cgroup limit and see it enforced, mount overlayfs
  (lower+upper+work), create a veth pair into a bridge across a net namespace, mount tmpfs,
  loop-mount an image file — each step prints PASS/FAIL. **Seccomp filtering is NOT smoke-tested
  here** (Alpine base ships no seccomp CLI and the rootfs build has no compiler for a probe
  helper); its runtime proof is relocated to E3.5-T03, whose runner installs a runc-style
  filter and asserts a blocked syscall returns EPERM. `SECCOMP`/`SECCOMP_FILTER` remain
  config-verified `=y` here.
- Native expect-test running the smoke script in a booted Alpine and asserting all-PASS.

## Acceptance criteria
- [ ] The smoke script passes every step in a booted guest (native, transcript in the log).
- [ ] `unshare -p -m -n -u -i -f --mount-proc sh -c 'echo NS_OK'` works (the T03 runner's
      exact shape).
- [ ] overlayfs mount over tmpfs upper + ext4 lower serves reads/writes/whiteouts correctly
      (create/delete/modify across layers).
- [ ] A memory cgroup limit demonstrably kills an over-allocating process (OOM in the
      group, guest stays alive).
- [ ] TUN + SQUASHFS each has a recorded decision (enabled + rebuilt, or deferred with the
      task that needs it named).

## Adversarial verification
The grep is not the proof — every "=y" claim tested here must be exercised in-guest (an
emulator missing a syscall makes the config a lie). Specifically hunt emulator gaps: clone3
with namespace flag combinations, setns across all types (the veth peer moves into a real
net ns and must ping across), pivot_root + umount2(DETACH) inside a mount namespace, and —
critically for the assert quality — no step may FALSE-PASS (report PASS while the capability
is broken). seccomp SECCOMP_SET_MODE_FILTER actually filtering (a blocked syscall returns
EPERM/kills) is proven in **E3.5-T03** (the runner installs the filter), not here. Any step
that fails IN THE GUEST but works on real riscv64 Linux is an emulator bug to file + fix,
and refutes until triaged.

**2026-07-06 — adversarial cold-clone critic on smoke-script ASSERT QUALITY (pass 2).** Charter:
"can any step FALSE-PASS (report PASS while the capability is broken)?" — the highest-value bug
class for a smoke test. Critic verdict FIX-FIRST; findings applied:
- **CRITICAL — cgroup OOM step false-passed three ways.** (a) `( echo $$ > cgroup.procs; exec … )`
  moved the WRONG process: in a POSIX subshell `$$` is the *parent script's* pid (critic REPRODUCED:
  parent==subshell==42435), so the memcg limit was never applied to the allocator — capability
  untested. (b) Keying PASS on a nonzero exit code counted a global-OOM/signal/error as success.
  (c) the O(n²) shell-string allocator would blow the interpreter budget. FIX: write the pid from
  *inside* the exec'd process (`sh -c 'echo $$ > …; exec tail /dev/zero'`), and assert the kernel's
  own `memory.events:oom_kill > 0` counter — immune to global-OOM false-passes. Also `-d`→`-f` on the
  `cgroup.controllers` availability guard (it's a FILE; `-d` was a latent false-FAIL on Alpine).
- **MAJOR — user-ns marker vacuous.** `unshare -U -r sh -c 'id -u' | grep '^0'` passes with or without
  a userns since the guest is root. FIX: assert the `/proc/self/ns/user` inode inside the unshare
  DIFFERS from the caller's — positive proof a distinct userns was created.
- **MAJOR — veth+bridge tested only root-netns plumbing.** The task wants the cross-netns setns
  datapath. FIX: move the peer into a real `ip netns` and `ping` across it — proves connectivity,
  not just enslavement.
- **MAJOR — seccomp was a named deliverable tested nowhere** (config-grep only, the exact "config is
  a lie until exercised" gap). Alpine base has no seccomp CLI + no compiler in the rootfs build, so
  RELOCATED formally to E3.5-T03 (runner installs a runc-style filter, asserts EPERM) — deliverable
  + AC edited here and added there so nothing is silently dropped.
- **LOW** — added a `pids >= 1` lower bound on the PID-ns check; regex `^[0-9]*$`→`^[0-9][0-9]*$`.
Critic also CROSS-CHECKED the config audit (accurate — all claimed `=y` confirmed, TUN/SQUASHFS
confirmed off) and the Rust harness (echo-proof + KillOnDrop sound). `bash -n` clean after fixes.
