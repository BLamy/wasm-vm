## Goal
Make the Docker tab visibly real before more hidden infrastructure lands: clicking **Run**
on the bundled busybox image boots the actual browser VM path, uses the bundled busybox/rootfs
artifact, runs one real command through the guest, and streams the real output back into the
Docker tab.

## Context
The previous Docker-tab pass removed the fake in-page shell and fake image catalog, but the
visible affordance still stops short of an undeniable run. This task is the product spine for
the next work: one button, one bundled image, one real guest command, one visible transcript.

This is deliberately smaller than the full OCI/database capstone. No registry pull-through,
no broad image matrix, no postgres, and no external networking are required here. If the
implementation needs a narrow runner shim or a pre-baked bundle mount to prove the path, do
that and document the non-claims. The only thing that may not be faked is command execution:
the output must come from the guest kernel/userspace path, not a JavaScript command
interpreter or canned transcript.

## Deliverables
- Docker-tab **Run** button for the bundled busybox image.
- Browser boot wiring that reuses the real `WasmLinux` / terminal boot path and attaches the
  Docker-tab output pane to the real console transcript.
- Bundled busybox/rootfs path available to the guest, with the exact artifact path and
  manifest digest shown in the UI.
- One default command that proves real execution, for example:
  `sh -lc 'echo CONTAINED_$((6*7)); uname -m; id'`.
- Playwright coverage for the visible path, including a screenshot or trace artifact that
  shows the Docker tab, the Run action, and the real command output.

## Acceptance criteria
- [ ] Fresh browser profile: open the Docker tab, select bundled busybox, click Run, and see
      a live boot transcript followed by a command prompt/output from the guest.
- [ ] The output contains a computed marker (`CONTAINED_42`, computed inside the guest),
      `uname -m` reports a RISC-V architecture, and the command exits 0.
- [ ] Removing or corrupting the bundled busybox/rootfs artifact makes Run fail with a typed
      visible error; it must not fall back to a mock shell or canned output.
- [ ] The UI clearly distinguishes this verified bundled busybox path from unbundled images
      that still require native pull/bundle work.
- [ ] Playwright asserts there is no fake command interpreter path left reachable from the
      Docker tab and that the visible output came through the real terminal/guest channel.

## Adversarial verification
Patch the browser bundle to remove the guest boot call, remove the busybox artifact, or
replace the guest command with a literal preloaded transcript; the Playwright test must fail
in each case. Disable arithmetic expansion in the command or echo a literal marker from the
host; the marker check must catch it. Run from a fresh profile with caches cleared so stale
terminal output cannot satisfy the assertion. The verifier should also inspect `web/docker.js`
for any surviving JS shell, canned command output, fake digest, or hidden success path.

### Pass 2 — acceptance EXECUTED headlessly; found + fixed a real "command never ran" bug (PR #NNN, stacked on #157)
**Actually ran the pass-1 acceptance** (`npx playwright test docker-busybox`, real cold WASM boots) — and it
**FAILED**: the guest booted and streamed output to the pane, but the one real command never ran
(`CONTAINED_42` timed out) and the terminal was dead to all further input, frozen at `~ #`. Pass 1 shipped
this because the spec had never been executed (code-review only). **This is exactly Brett's "I can't boot my
docker container / can't type" report — now reproduced and root-caused.**

**Root cause (re-entrant machine call), proven by instrumentation, not guessed:** `docker.js` typed the
command from inside the `onConsole` callback. That callback runs **synchronously inside the guest's output
emission** — `machine.runChunk → onOutput → emitConsole → onConsoleChunk → maybeInject → injectCommand →
sendInput → machine.sendInput`. So `sendInput` re-entered the wasm machine **while `runChunk` was still on
the stack**, which the machine explicitly forbids. Captured trace:
`sink fired … reentrant=true → sendInput THREW: "re-entrant call into WasmMachine (a console callback
cannot drive the machine)"`. **Two effects, both required to explain the freeze** (mechanism corrected by
the cold-clone critic — my first draft wrongly said the throw "stopped the run loop"; it does not, because
`emitConsole` already wraps each subscriber in try/catch on the base branch, so the throw is swallowed and
the run loop keeps ticking): (1) the command was dropped, and because `injectCommand` had already set
`injected=true` and cleared its fallback timer, it is **never retried** → the prompt stays bare; (2) the
throw, unwinding **through the terminal's `pump()`**, skips the line that resets `draining=false`, so
`draining` sticks `true` and the JS input queue **jams permanently** → every later keystroke is enqueued
but never drained (this is why, in the diagnostics, a separate non-re-entrant `sendInput` AFTER the
re-entrant one also failed until a sink-level guard was added). Net user-visible symptom: guest frozen at
`~ #`, terminal dead to all input.

**Isolation matrix (headless A/B/C/D) that pinned it:** `/`+`#boot-linux` ✅, `/#docker`+`#boot-linux` ✅,
`/`+direct `runBusybox` ✅, `/#docker`+**Docker Run button** ❌ — i.e. neither navigation nor the boot
trigger, only the docker-run path (which drives input from the console callback) broke input.

**Fix:**
- `web/docker.js`: `injectCommand` defers its `sendInput` to a fresh macrotask (`setTimeout(…,0)`) so it
  lands AFTER `runChunk` returns — never re-entrant.
- `web/main.js`: the input sink now contains a stray re-entrant/failed `sendInput` (try/catch, logged at
  `error` level) so a future callback-driven caller can't jam the terminal's input queue — and the
  Playwright console-error gate would catch such a regression instead of it failing silently.

**Adversarial cold-clone critic (static/call-graph, rr unavailable on macOS): verdict CLEAN.** Confirmed the
mechanism in Rust (`crates/wasm/src/lib.rs`: `run_chunk` holds `try_borrow_mut` across the synchronous JS
output callback; any re-entry → `reentrant()` JsError). `setTimeout(0)` runs on an empty stack after the
borrow drops — sufficient; `send_input` only appends to a Vec (never calls `run_chunk`), and `pump` calls
`sink` at most once synchronously — no remaining synchronous re-entry path. Idempotency (`injected` +
`clearTimeout`) blocks double-send; the deferred closure captures a fresh `cmd` (not `session`), so a
re-render/reboot can't send a stale command at the machine level. Completeness: the only output callback is
`onOutput → emitConsole → onConsoleChunk`; `term.onData`/Fit are DOM events, not callbacks — covered.
**Findings folded in:** corrected the "stopped the run loop" narrative (above); raised the sink guard log
from `warn` to `error` so the test error-gate catches future callback-driven-input regressions.

**Verified headlessly:** `web/tests/docker-busybox.spec.js` — **1 failed → 4 passed** across the fix
(real `CONTAINED_42` computed in-guest via `$((6*7))`, `uname -m`=riscv64, `WVM_EXIT_0`; plus the
