# Container matrix — the OCI pipeline runs any riscv-compatible image (E3.5-T04b)

Evidence that `sideload → oci unpack → bundle` is **generic**, not postgres-specific. Each
image is pulled (digest-verified), unpacked, and statically checked. **A PASS means ALL of:**
unpack succeeds; `run.json` argv[0] resolves to a REAL regular executable in the rootfs (via
absolute path / the image PATH / symlinks CLAMPED inside the rootfs — a dangling/looping/
escaping symlink or a directory is NOT resolved); that target is arch-correct (an ELF is
RISC-V 64-bit; a `#!`-script's interpreter, including the `env <prog>` argument, resolves to a
RISC-V 64-bit ELF); and **arch purity over the WHOLE rootfs** — every ELF anywhere (no dir
restriction, no truncation) is RISC-V 64-bit. One foreign binary anywhere fails the image.

This check is **adversarially self-tested** (`tools/oci-matrix-selftest.sh`): it reproduces the
four false-passes the E3.5-T04b critic found in the first version — a dangling-symlink
entrypoint, a foreign server binary OUTSIDE the standard bin dirs (postgres' real `postgres`
binary lives in `/usr/lib/postgresql/18/bin`, now scanned — hence 917 ELF, not 294), a foreign
ELF past the old 400-file scan cap, and a rootfs-escaping symlink — and confirms each is now
REJECTED. So a PASS is not vacuous. Pulling alpine for **amd64** also FAILs (x86-64 ELFs).

## riscv64 availability (Docker Hub survey, 2026-07-06)

**Publish riscv64:** alpine, busybox, nginx, httpd, caddy, haproxy, redis, memcached,
postgres, traefik, debian, ubuntu, python, ruby, rust, golang.
**No riscv64 yet:** node, mariadb, mysql, mongo — `oci-sideload` fails cleanly ("no riscv64
manifest in the index", non-zero exit), never a crash.

## Deferred: the boot leg

Actually `wvrun`-ing each image and watching it serve is boot-gated — it needs the bundle-
into-guest mechanism (a second virtio-blk disk carrying the bundle rootfs) + a long
interpreter boot the CI env kills. That is the E3.5-T05 capstone leg.
